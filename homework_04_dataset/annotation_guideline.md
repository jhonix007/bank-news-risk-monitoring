# Инструкция по разметке

## Единица разметки

Одна строка = один фрагмент новости вокруг одного найденного банка.

## Поля ручной разметки

- `risk_type`
- `entity_relevance`
- `alert_flag`
- `alert_reason`
- `label_quality`
- `review_comment`, если используется assisted-файл

`risk_score_v1` вручную не размечается. Он рассчитывается после проверки разметки.

Автоматическая разметка в полях `*_suggested` является только предразметкой. Финальная разметка считается готовой только после ручной проверки человеком и заполнения финальных полей `risk_type`, `entity_relevance`, `alert_flag`, `alert_reason`, `label_quality`.

## risk_type

- `no_risk` — риска нет.
- `operational_issue` — операционный сбой.
- `data_leak_security` — утечка данных / информационная безопасность.
- `fraud_phishing` — мошенничество / фишинг.
- `legal_regulatory` — суд / штраф / расследование / регуляторика.
- `sanctions` — санкции.
- `customer_complaints` — массовые жалобы клиентов.
- `other_risk` — другой риск.

## entity_relevance

- `direct` — риск относится именно к найденному банку.
- `indirect` — риск относится к рынку или общему контексту.
- `mentioned_only` — банк просто упомянут.
- `unclear` — непонятно, нужна проверка.

## alert_flag

- `1` — показать карточку специалисту.
- `0` — не показывать.

Правило: `alert_flag = 1`, если `risk_type != no_risk`, `entity_relevance = direct` и публикация требует внимания специалиста.

## alert_reason

Короткое объяснение, почему публикация попала в карточку.

Пример: "В тексте говорится о массовом сбое переводов у клиентов ВТБ."

## label_quality

- `ok` — уверенная разметка.
- `ambiguous` — спорный пример.
- `need_review` — нужна повторная проверка.

## Ложные совпадения по банкам

Иногда найденный алиас может не быть банком. Например, слово "открытие" может относиться к открытию мероприятия, а не к банку.

Если разметчик видит, что строка вообще не про банк:

- `risk_type = no_risk`
- `entity_relevance = unclear`
- `alert_flag = 0`
- `label_quality = need_review`
- в `review_comment` написать: "ложное совпадение: не банк"

Такие строки лучше удалять из финального датасета после проверки. Скрипт финализации автоматически сохраняет их отдельно в `data/processed/rejected_after_review.csv`, если `label_quality = need_review` и `review_comment` содержит "ложное совпадение".

## Примеры

| Ситуация | risk_type | entity_relevance | alert_flag | alert_reason | label_quality |
|---|---|---|---:|---|---|
| В новости Сбербанк упомянут как участник форума, риск-событий нет. | `no_risk` | `mentioned_only` | 0 |  | `ok` |
| В тексте говорится о сбое переводов у клиентов ВТБ. | `operational_issue` | `direct` | 1 | Массовый сбой переводов у клиентов ВТБ. | `ok` |
| Альфа-Банк упомянут в списке банков, но риск относится к другому участнику рынка. | `no_risk` | `mentioned_only` | 0 |  | `ok` |
| В статье описан штраф другого банка, а Т-Банк только сравнивается с ним. | `legal_regulatory` | `indirect` | 0 |  | `ambiguous` |
| В новости сообщается о новых санкциях против Газпромбанка. | `sanctions` | `direct` | 1 | Санкции напрямую относятся к Газпромбанку. | `ok` |
| Клиенты массово жалуются на недоступность приложения Совкомбанка. | `customer_complaints` | `direct` | 1 | Массовые жалобы клиентов на приложение Совкомбанка. | `ok` |
| По фрагменту неясно, относится ли утечка данных к найденному банку. | `data_leak_security` | `unclear` | 0 |  | `need_review` |
