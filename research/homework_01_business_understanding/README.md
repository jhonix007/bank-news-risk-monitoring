# Домашнее задание №1. Бизнес-анализ проекта

## Проект

**AI-модуль выявления и приоритизации банковских риск-инфоповодов на основе NER, entity matching и классификации риск-категорий**

Краткое описание:

> MVP NLP-сервиса, который анализирует новостной поток, находит упоминания банков, сопоставляет найденные названия с единым справочником организаций, определяет тип риск-инфоповода и рассчитывает explainable risk score для приоритизации карточек в дашборде.

---

# 1. Бизнес-анализ / Business Understanding

## 1.1. Цели и скоуп проекта

### Бизнес-проблема

Банки работают в информационно чувствительной среде. Новостные публикации о сбоях, утечках данных, мошенничестве, судебных и регуляторных событиях могут быстро стать репутационным или операционным риском.

Проблема не только в том, чтобы найти упоминание банка. Важно понять:

```text
1. Какой банк упомянут?
2. Это обычная новость или риск-инфоповод?
3. Какой это тип риска?
4. Относится ли риск именно к этому банку?
5. Насколько срочно нужно реагировать?
6. Кому внутри банка передать сигнал?
```

Пример:

```text
"Сбербанк прокомментировал сбой в приложении ВТБ"
```

Простое keyword-правило может ошибочно связать слово «сбой» со Сбербанком, хотя риск относится к ВТБ. Поэтому в проекте используется entity-centric подход:

```text
сначала найти банк
→ затем сопоставить его со справочником
→ затем определить риск-контекст
→ затем рассчитать приоритет
```

### Бизнес-цель

**Цель проекта** — разработать MVP-модуль, который помогает PR- и риск-командам банка быстрее выявлять и приоритизировать риск-инфоповоды в новостном потоке.

Бизнес-цель можно сформулировать так:

> Сократить время первичного анализа новостного потока за счёт автоматической сортировки инфоповодов по банку, типу риска и уровню приоритета.

### Что входит в скоуп MVP

В рамках MVP планируется реализовать:

- загрузку исторического новостного датасета;
- поиск упоминаний банков;
- entity matching: сопоставление разных вариантов названия банка с единым справочником;
- классификацию фрагмента новости по типу риска;
- сравнение нескольких подходов к классификации;
- расчёт explainable risk score;
- Streamlit-дашборд с карточками риск-инфоповодов;
- логирование экспериментов в MLflow;
- упаковку приложения в Docker.

### Что не входит в скоуп MVP

В рамках первой версии проекта не планируется:

- промышленный сбор новостей из тысяч источников;
- скрейпинг СМИ, Telegram и соцсетей;
- конкуренция со СКАН, Медиалогией или Brand Analytics по охвату источников;
- полноценная интеграция с внутренними банковскими системами;
- автоматическое принятие решений без участия человека.

Источник новостей рассматривается как заменяемый компонент. В учебном MVP используется открытый исторический датасет, а в промышленном сценарии источником может быть API, RSS, выгрузка из системы медиамониторинга или внутренний поток банка.

### Потенциальный заказчик

В рамках учебного проекта реальный заказчик отсутствует. Потенциальным заказчиком может быть банк или финансовая организация.

Возможные бизнес-заказчики:

- департамент внешних коммуникаций;

- подразделение операционных рисков;
- служба безопасности / антифрод;
- юридический департамент / compliance.

### Основной пользователь

Основной пользователь MVP:

**PR-аналитик или риск-менеджер банка.**

Его задача — не читать сотни новостей вручную, а быстро получить список инфоповодов, которые требуют внимания.

Пользователь должен увидеть:

- какой банк упомянут;
- какой риск найден;
- насколько инфоповод приоритетный;
- почему система считает его рискованным;
- кому внутри банка нужно передать сигнал.

### Дополнительные пользователи

| Руководитель направления | краткий приоритизированный риск-дайджест |

Прямых рабочих чатов с потенциальными пользователями пока нет. Это ограничение будет компенсировано:

## 1.3. Текущая ситуация / Assessing Current Solution

### Регуляторный контекст

Банк России рассматривает риск потери деловой репутации как значимый риск кредитной организации. В письме № 92-Т среди факторов риска указана публикация негативной информации о кредитной организации, её служащих, учредителях, органах управления и связанных лицах в СМИ.

Также Банк России рекомендует кредитным организациям определять порядок:

- выявления факторов риска потери деловой репутации;
- оценки риска;
- мониторинга риска;
- информирования органов управления и подразделений.

Это подтверждает, что задача выявления негативной информации и репутационных факторов имеет регуляторный контекст.

### Существующие решения

На рынке уже существуют промышленные системы медиамониторинга:

- СКАН-Интерфакс;
- Brand Analytics;
- Медиалогия;
- YouScan;
- Babkee.

Их сильные стороны:

- широкий охват источников;
- мониторинг СМИ, соцсетей, Telegram, форумов и отзывов;
- отчётность;
- PR-аналитика;
- оповещения;
- API и корпоративные сценарии.

### Почему проект всё равно имеет смысл

Проект не конкурирует с промышленными системами как поставщик данных.

Позиционирование проекта:

> AI-модуль анализа и приоритизации, который может быть подключён поверх внешнего источника новостей.

Отличия MVP:

| Критерий | Промышленные системы | Учебный MVP |
|---|---|---|
| Источники | широкий сбор данных | источник заменяемый |
| Фокус | медиамониторинг и PR-аналитика | банковские риск-инфоповоды |
| Основная ценность | охват, отчёты, мониторинг | NER + matching + risk-type classification |
| Тип результата | поток упоминаний и аналитика | приоритетные карточки риска |
| Прозрачность | зависит от платформы | explainable risk score |

## 1.4. Данные и ресурсы

### Данные

Для MVP планируется использовать открытый исторический новостной датасет:

- **Lenta.ru News Dataset**;
- либо **RuNews**.

В MVP новостной поток будет симулироваться по дате публикации.

Объектом анализа будет не вся статья целиком, а фрагмент текста вокруг упоминания банка:

```text
заголовок + 1–2 предложения вокруг найденного банка
```

Это снижает шум и упрощает разметку.

### Почему не live scraping

Скрейпинг СМИ, соцсетей и Telegram создаёт юридические и технические риски. Поэтому для первого этапа используется готовый открытый датасет.

В промышленном сценарии источник может быть заменён на:

- RSS;
- API новостей;
- GDELT / NewsAPI;
- выгрузку из СКАН / Медиалогии / Brand Analytics;
- внутренние данные банка;
- сообщения клиентского сервиса.

### Технические ресурсы

Доступные ресурсы:

- Python;
- Jupyter Notebook;
- pandas, numpy, scikit-learn;
- NLP-библиотеки для русского языка;
- transformers / rubert-tiny при наличии ресурсов;
- MLflow для логирования экспериментов;
- Streamlit для MVP-дашборда;
- Docker для упаковки приложения;
- GitHub/GitLab для хранения кода и отчётов.

Дополнительная закупка железа не требуется. Для baseline-моделей достаточно CPU. Для rubert-tiny желательно наличие GPU, но это не является обязательным условием для MVP.

### Экспертная поддержка

Прямого доступа к PR-аналитикам или риск-менеджерам банков на текущем этапе нет.

Экспертная поддержка доступна через:

- консультации с преподавателем;
- обратную связь от эксперта по проекту;
- анализ открытых материалов Банка России;
- анализ конкурентов;
- оценку качества MVP на размеченной выборке.

## 1.6. Решаемые задачи с точки зрения аналитики / Data Mining Goals

### Основная аналитическая постановка

Основная ML-задача проекта:

> классификация фрагмента новости по типу банковского риска.

Формально:

```text
text_fragment → risk_type
```

Где `risk_type` — один из классов:

```text
0. нет риска
1. операционный сбой
2. утечка данных / информационная безопасность
3. мошенничество / фишинг
4. суд / штраф / расследование
5. санкции / регуляторика
6. массовые жалобы клиентов
```

### Дополнительные аналитические компоненты

#### 1. NER / поиск банков

Задача:

> найти в тексте упоминание банка или организации.

Пример:

```text
"Клиенты Сбера пожаловались на сбой переводов"
→ найдено: "Сбера"
```

#### 2. Entity matching

Задача:

> сопоставить найденное название с единым справочником банков.

Пример:

```text
"Сбера" → "Сбербанк"
"СберБанк Онлайн" → "Сбербанк"
"Тинькофф" → "Т-Банк"
```

#### 3. Risk-type classification

Задача:

> определить тип риска по фрагменту новости.

Пример:

```text
"Клиенты Сбербанка пожаловались на сбой переводов"
→ операционный сбой
```

#### 4. GPT zero-shot baseline

GPT/LLM будет использоваться не как единственный источник истины, а как отдельный baseline:

```text
текстовый фрагмент → GPT-классификация risk_type
```

Результат GPT будет сравниваться с классическими моделями на вручную проверенной выборке.

#### 5. Risk score

Задача:

> рассчитать итоговый уровень риска для пользователя.

Risk score будет объяснимой формулой, а не отдельной black-box моделью.

Возможные факторы:

- тип риска;
- уверенность модели;
- связь риска с найденным банком;
- негативность формулировки;
- значимость источника;
- динамика инфоповода, если доступна.

## 1.7. Метрики качества и критерии успеха

### Метрики для NER / matching

| Метрика | Смысл |
|---|---|
| Precision | доля корректно найденных банков среди всех найденных |
| Recall | доля найденных банков среди всех реально присутствующих |
| F1-score | баланс precision и recall |

### Метрики для классификации risk_type

| Метрика | Смысл |
|---|---|
| Accuracy | общая доля правильных классификаций |
| Precision macro | качество по классам без учёта дисбаланса |
| Recall macro | полнота по классам без учёта дисбаланса |
| F1 macro | основная метрика сравнения моделей |
| F1 по каждому классу | контроль качества по отдельным типам риска |
| Confusion matrix | анализ ошибок между классами |

### Продуктовые метрики

| Метрика | Смысл |
|---|---|
| Precision@K | сколько релевантных карточек попало в топ-K |
| False alert rate | доля ложных алертов |
| Coverage по банкам | сколько банков из справочника система умеет находить |
| Coverage по risk_type | сколько риск-классов система покрывает |
| Время ручного просмотра | proxy-оценка экономии времени аналитика |

### Критерии успешности MVP

Минимальный успех:

- подготовлен справочник 5–7 банков и алиасов;
- подготовлены 6–7 риск-категорий;
- размечено минимум 300–500 фрагментов;
- реализован rule-based baseline;
- обучена хотя бы одна ML-модель: TF-IDF + Logistic Regression;
- проведено сравнение моделей через MLflow;
- рассчитаны F1 macro, precision, recall, confusion matrix;
- реализован Streamlit-дашборд;
- проект упакован в Docker.

Ориентировочные пороги качества:

| Компонент | Минимальный успех | Хороший результат |
|---|---:|---:|
| Entity matching F1 | 0.80 | 0.90+ |
| Risk-type F1 macro | 0.60 | 0.70+ |
| Precision@10 карточек | 0.60 | 0.75+ |

Значения являются предварительными и будут уточняться после анализа данных и первой итерации моделирования.

## 1.8. Продуктовая гипотеза и импакт

### Продуктовая гипотеза

Если автоматически находить банк, определять тип риск-инфоповода и ранжировать карточки по risk score, то PR- или риск-аналитик сможет быстрее выделять новости, требующие реакции.

### Ожидаемый импакт

Ожидаемый эффект MVP:

- снижение времени первичного просмотра новостей;
- уменьшение количества нерелевантных упоминаний;
- повышение вероятности раннего обнаружения риск-инфоповода;
- более понятная маршрутизация сигнала внутри банка.

### Как оценивать импакт в учебном MVP

Так как нет доступа к реальным пользователям, импакт будет оцениваться через proxy-метрики:

- сколько новостей система отфильтровала как нерисковые;
- сколько релевантных карточек попало в top-K;
- сколько ложных алертов появилось;
- насколько корректно определён банк и тип риска;
- насколько карточка понятна с точки зрения причины risk score.

## 1.9. Обратная связь от пользователей и рынка

На текущем этапе прямой обратной связи от конечных пользователей нет. Это ограничение проекта фиксируется явно.

Так как нет доступа к PR-аналитикам или риск-менеджерам банков, пользовательская ценность будет проверяться через косвенные источники:

### 1. Регуляторный контекст

Банк России рассматривает риск потери деловой репутации как значимый риск кредитной организации и рекомендует определять порядок его выявления, оценки и мониторинга.

### 2. Анализ конкурентов

Наличие промышленных систем СКАН-Интерфакс, Brand Analytics, Медиалогия и других решений подтверждает, что рынок нуждается в инструментах мониторинга и анализа публичной информации.

### 3. Экспертная консультация

В рамках консультации с преподавателем было предложено сузить задачу до NER + matching + риск-контекста, чтобы избежать субъективной разметки "кризис / не кризис".

### 4. Proxy-evaluation

На первом этапе вместо прямого пользовательского тестирования будет применена proxy-оценка:

- правильно ли определён банк;
- правильно ли определён тип риска;
- релевантна ли карточка инфоповода;
- попадают ли важные карточки в top-K;
- понятны ли причины risk score.

### 5. План будущей обратной связи

На следующей итерации можно провести 3–5 коротких интервью с представителями PR, риск-менеджмента, клиентского сервиса или банковского аналитического подразделения.

Возможные вопросы:

- Какие типы инфоповодов наиболее критичны?
- Какие ложные срабатывания неприемлемы?
- Достаточно ли объяснения risk score?
- Кому внутри банка должен уходить алерт?
- Какой формат удобнее: дашборд, email, Telegram, API?

Корректная формулировка для защиты:

> На данном этапе прямой обратной связи от конечных пользователей нет, так как проект находится в стадии учебного MVP. Потребность подтверждается регуляторным контекстом, существованием промышленных решений медиамониторинга и экспертной консультацией. Пользовательская ценность будет оцениваться через proxy-evaluation: релевантность карточек, точность определения банка, типа риска и качество приоритизации top-K инфоповодов. В следующей итерации планируется получить обратную связь от представителей PR- или риск-функций.

## 1.10. План проекта / Project Plan

Хотя в рамках текущего домашнего задания заполняется только бизнес-анализ, в шаблоне требуется описать план всех шести фаз проекта. Ниже приведён план работ без детального заполнения следующих разделов.

| Фаза | Что будет сделано | Результат |
|---|---|---|
| 1. Business Understanding | постановка бизнес-проблемы, ЦА, цели, риски, метрики | текущий Markdown-документ |
| 2. Data Understanding | загрузка Lenta.ru / RuNews, EDA, анализ структуры данных, длины текстов, частоты банков и риск-слов | EDA-ноутбук |
| 3. Data Preparation | очистка текстов, выделение фрагментов вокруг банков, словарь банков, словарь рисков, разметка 300–500 примеров | подготовленный датасет |
| 4. Modeling | rule-based baseline, TF-IDF + LogReg, TF-IDF + SVM, rubert-tiny, GPT zero-shot baseline | MLflow-эксперименты |
| 5. Evaluation | F1 macro, precision/recall, confusion matrix, error analysis, Precision@K | отчёт по качеству |
| 6. Deployment | Streamlit-дашборд, risk score, Docker, README с запуском | MVP-приложение |

### Усиленный технический план

```text
Lenta.ru / RuNews
        ↓
EDA:
- длина текстов
- частота банков
- распределение риск-категорий
- частые слова по классам
        ↓
NER / словарь банков
        ↓
Entity matching:
"Сбер", "Сбера", "СберБанк Онлайн" → "Сбербанк"
        ↓
Предварительная разметка:
- словарь риск-категорий
- GPT-assisted labeling
- ручная проверка части примеров
        ↓
Моделирование + MLflow:
1. rule-based baseline
2. TF-IDF + Logistic Regression
3. TF-IDF + SVM
4. rubert-tiny
5. GPT zero-shot baseline
        ↓
Evaluation:
- F1 macro
- precision / recall
- confusion matrix
- error analysis
        ↓
Risk score:
- тип риска
- уверенность модели
- связь с банком
- негативность
        ↓
Streamlit dashboard
        ↓
Docker
```
